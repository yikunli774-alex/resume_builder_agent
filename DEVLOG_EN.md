## 2026-05-31 ~ 06-01 — Tasks 1–5 implemented

### Pitfall: MongoDB connection fails with `'X509' object has no attribute 'get_extension'`
- **Symptom**: testing `save_resume_version` threw `ServerSelectionTimeoutError`; Atlas connection timed out after 30s.
- **Cause**: `service_identity` was still the 2018 release (18.1.0), which calls `X509.get_extension` — a method newer `pyOpenSSL` versions have removed. Not a bug in my code.
- **Fix**: `python -m pip install --upgrade service_identity` (18.1.0 → 26.1.0).
- **Lesson**: nine times out of ten, Python TLS/SSL errors are dependency versions fighting each other. Check package versions first; don't rush to change your own code.

### Pitfall: collection name typo in mongo_tools
- `save` wrote into `"resume_vesion"` (missing letter, singular), but `list`/`load` read `"resume_versions"` → writes were invisible to reads.
- Unified to `"resume_versions"`.

### Decision: Gemini keeps wrapping JSON in markdown code fences, breaking `json.loads`
- Two options: A) set `response_mime_type="application/json"` on the model (fixes the root cause); B) strip the ```` ```json ```` wrapper in code (treats the symptom).
- **Chose A** — output is clean JSON and the parsing code stays untouched.

### Decision: suggestion ordering is done in code, not by the model
- Even when explicitly told to sort by `impact_score` descending, the model frequently doesn't.
- `.sort(key=..., reverse=True)` in code as the backstop — deterministic logic should never be delegated to the model.

### Pitfall: playwright installed but won't run
- After `pip install playwright` you still need `python -m playwright install chromium` to download the browser binary (like Selenium needing a driver).

### Environment lesson: always `python -m pip`, never bare `pip`
- Bare `pip` and `/opt/anaconda3/bin/python` are not the same environment; packages land in the wrong place.

### check_formatting edge handling (Task 5)
- Use `.get(k, default)` instead of `obj[k]` so missing fields don't crash.
- One outer try/except catches malformed input and returns `internal_error`, but each rule is independent inside, so one blown rule doesn't stop the rest.

## 2026-06-01 — Task 6 `rewrite_bullet.py` done

### Pattern: Generator–Validator (generate, validate, self-correct loop)
- `rewrite_bullet` calls Gemini to generate → `_check_bullet_rubric` validates in pure code → on failure the violations are appended to the next prompt, up to MAX_RETRIES (3); returns immediately on success.
- **Why this is not wasteful**: ① validation is local code, costs no LLM call; ② early exit on success — good bullets measured attempts=1, not 3; ③ granularity is a single sentence, not the whole document. 3 is a ceiling, not the norm.
- Measured: `"Helped with search feature"` → `"Optimized search ... by 15%"`, attempts=1, passed first try.

### Pitfall: action-verb check is brittle against punctuation
- `bullet.split()[0]` can return a first word with punctuation attached (`"-"`, `"Reduced,"`, `'"Reduced'`) which never equals anything in ACTION_VERBS → false "no action verb" → pointless retries.
- **Fix**: strip non-letters with `re.sub(r"[^a-zA-Z]", "", word)` before comparing.

### Decision: drive the weak-phrase check from yaml
- Added a `no_weak_phrase` check reading the `bad` list from rubric.yaml ("Worked on" / "Responsible for" …); strips leading symbols then checks the opening words.
- `_load_bad_phrases()` is wrapped in try/except — if the file is unreadable it degrades to an empty list and never drags down other checks (same backstop philosophy as check_formatting).
- Bonus: the previously dead `yaml` / `RUBRIC_PATH` imports finally earn their keep.

### Open question: rubric's has_quantification is a hard requirement
- Unquantifiable bullets (e.g. "Collaborated with design team on UX") can't gain real numbers in 3 attempts → always run the full 3 retries and end marked failed. MAX_RETRIES prevents infinite loops, but the calls are wasted.
- Whether to relax it ("has a number **or** a concrete technology name" passes) is a product decision. Pending.

### Architecture understanding: tool arguments are generated by Gemini, delivered by ADK
- `original_bullet`/`instruction`/`context` are never assigned anywhere in the code — Gemini "fills in the form" at runtime based on the docstring, and the ADK framework passes them into the function. The docstring is Gemini's tool manual.
- Note the two different `instruction`s: `LlmAgent(instruction=)` is the agent's system prompt; `rewrite_bullet(instruction=)` is a per-call argument. Same name, different things.

## 2026-06-03 — Task 9 `adk web` integration pitfalls

### Model choice: orchestrator pro vs flash trade-off
- All three agents started on `gemini-2.5-pro` → 13–21s per LLM call, and one "parse" stacks 2–3 calls — slow enough to look frozen.
- Switching everything to `flash` was fast (2–4s), but **the orchestration layer on flash produced `MALFORMED_FUNCTION_CALL`**: analyze_jd_match had to inline the whole resume_json + the whole JD into one function call, and flash corrupted the large, deeply structured JSON.
- **Current compromise**: root_agent (orchestrator) on `pro` (more reliable at assembling big calls), the two search sub-agents stay on `flash`. Pro is slow but stable.
- **The real cure (todo) = session state**: store resume_json in ADK session state and have downstream tools read it from there instead of passing it as a function-call argument. Big data stays out of function calls, so even flash can't corrupt it. It's a medium refactor (4–5 tool signatures + docstrings, deviating from ASSIGNMENT's original signatures); deferred.

### Pitfall: LLM suggestions drift outside the schema (agent keeps proposing a professional summary)
- Symptom: analyze_jd_match repeatedly suggested "add a professional summary", but the resume schema has no such field and the template has nowhere to render it → unactionable (empty) suggestions.
- Root cause: the Gemini inside the tool is a "generic resume consultant" speaking from common sense — **it doesn't know which fields our schema supports**. Generic model knowledge ≠ operations the system can execute; a classic LLM-agent disconnect.
- **Fix (done)**: constrain the analyze_jd_match prompt — suggestions may only target schema-supported sections (personal_info/education/experience/projects/skills/additional) and existing bullets; explicitly forbid summary/objective/cover-letter suggestions. **Muzzle the model so its suggestions are executable.**
- Decision: do **not** build a summary feature (student resumes don't need one; adding it is feature creep).

### Pitfall: LLM suggestions are timid, won't cut, and self-rationalize
- Symptom: for an AI-direction role, the agent still suggested **keeping and polishing** low-level C systems projects (cache/heap/malloc), never suggesting removal or demotion, and claimed they "reflect observational skills" — textbook hallucinated rationalization.
- Root causes: ① analyze_jd_match is fundamentally a **keyword diff** (covered/missing) — tactical view only (polish bullets), no strategic view (is the project portfolio pointing the right way); ② the prompt frames the task as "improvement suggestions" → model biases to add/rewrite and almost never uses remove/reorder; ③ LLMs are inherently people-pleasing and reluctant to disown what the user already built.
- **Fundamental limitation**: strategic career advice is exactly where LLMs are weakest and most platitudinous. Prompting helps (add "domain relevance, dare to suggest removal" framing) but can't cure it.
- It also violates our own design principle "agent suggests; user decides" — keeping or cutting a project is the **human's** strategic call; the agent should present facts, not decide for the user and then congratulate itself.
- Decision: strategic judgment goes to the user; the agent is positioned as a tactical polisher. Whether to tune the prompt so it dares to suggest cutting off-target projects: pending.

### Red-line pitfall: hallucinated user experience (invented "used LangChain")
- Symptom: the user described an experience vaguely; the agent fabricated a concrete tech stack (LangChain) into the bullet.
- Nature: **resume fraud** — worse than not having the feature (collapses at the first interview question + an integrity problem). Root cause: the LLM instinct of "better to invent something plausible than leave a blank", compounded by rewrite's implicit goal of "more impressive" — truthfulness and impressiveness were fighting, and the design erred on the wrong side.
- **Fix (done)**: ① rewrite_bullet prompt gains a "CRITICAL TRUTHFULNESS RULE": only facts present in the input; no invented technologies/numbers/frameworks. ② agent instruction GUIDING PRINCIPLES leads with "TRUTHFULNESS IS ABSOLUTE": ask when information is missing, never fabricate.
- **Honest caveat**: prompts can largely suppress but **never 100% eliminate** hallucination. The real defense is the human reviewing every line ("agent suggests; user decides", again).

### Pitfall: fields like dates get lost as data moves between tools
- Symptom: after several agent turns, some date fields disappeared from the resume.
- Root cause: same as schema drift — resume_json is passed around as a function-call argument and regenerated by the model, which doesn't faithfully preserve fields. Same disease as the text/institution key drift.
- The real cure is still session state (deferred). Temporary mitigation: emphasize "preserve all input fields" in the prompt — unreliable.

### Insight: these prompt changes are context engineering / behavioral steering
- Changing instructions/constraints = steering model behavior with system text — **probabilistic steering**, not deterministic control (unlike if/else in code).
- For deterministic workflow enforcement → split PHASEs into sub-agents with a state machine (big change).
- Improvements **can't be quantified from a few manual runs**: needs an eval harness (metrics + test cases + before/after violation rates). Current sample size ~1; the honest claim is only "constrained multi-turn agent behavior via system instructions", not "improved X%".

### Integration bug: sync Playwright crashes inside adk web's async loop
- Symptom: after a successful save, PDF generation throws `You are using Playwright Sync API inside the asyncio loop. Please use the Async API instead.`
- Root cause: render_template used Playwright's **sync API** (sync_playwright), but adk web is an **asyncio server**. Playwright hard-forbids the sync API inside a running event loop (it would block it). Standalone scripts have no event loop, so unit tests passed; adk web exploded — classic "unit tests pass, integration reveals".
- **Fix (option A, subprocess isolation)**: extracted the HTML→PDF step into `_html_to_pdf()` and run sync Playwright in a clean `subprocess` (no event loop in the child, sync API fine). Main function logic untouched; try/except backstop added.
- **Verification**: reproduced the adk web environment by calling render_template inside `asyncio.run()` → after the fix, error=None and the PDF generated.
- Option B (switch to async_playwright + async def) is more orthodox but touches more (function signature and test approach); option A for the hackathon.

### Todo: schema drift
- Observed parse_resume emitting bullet key `text` (spec/downstream rewrite_bullet expect `content`) and education key `institution` (spec says `school`). Model free-styling causes key drift; downstream may not match. To be aligned.

## 2026-06-03 (evening) — session state refactor (phase 1, written, pending end-to-end verification)

### Motivation
MALFORMED + schema drift share one root cause: resume_json passed repeatedly as a function-call argument / rewritten by the agent. Solution = a single copy of the resume per session in ADK's `tool_context.state`; tools read from state; the agent no longer passes or rewrites it.

### Key mechanics (confirmed by research)
- ADK injects the parameter via the **type annotation** `tool_context: ToolContext`, and it is **invisible to Gemini** (not a parameter the model fills). So once a tool has it, the model never passes the resume at all.
- `tool_context.state` is a delta-aware dict; just read/write `state['resume_json'] = ...`.

### What changed (5 tools + agent.py)
- parse_resume: gains `tool_context`; on success `state['resume_json'] = resume_json` (the single write point / entry).
- analyze_jd_match / check_formatting / render_template / save_resume_version: **resume_json parameter removed**; read from `state.get('resume_json')`; graceful "No resume found in session" if absent.
- Technique: check/render bodies untouched — just bind the state value to a local variable of the same name at the top.
- agent.py: AVAILABLE TOOLS signatures updated (no resume passing) + fixed stale mongodb.* names; added a SESSION STATE section explicitly forbidding the agent from reconstructing/inlining the full resume.

### Verification
- Offline unit test (fake tool_context, .state=dict, run inside asyncio.run to simulate adk web): parse→analyze (score 68)→check (passed)→save (version_id)→render (PDF contains project names) — **whole chain green**, no tool received a resume_json argument anywhere; empty ctx degrades to "No resume found". ✅
- **Todo**: re-test end-to-end under adk web (next time) — confirm save+render double call no longer MALFORMEDs and the PDF doesn't drift.
- Note: phase 1 does not include "rewrite results written back to state" (edit write-back is phase 2). rewrite is still a standalone bullet-in/bullet-out function; rewritten content is not yet merged into the resume in state.

### Concept: why "carrying resume_json around" is slow — the context snowball
- **LLMs have no memory.** Each turn, the model knows exactly one thing: the full conversation history re-sent to it this time.
- So once resume_json enters the conversation history, it is **re-sent to the model every single turn** — a snowball — especially when the agent also keeps rewriting the whole resume, stacking multiple copies in context.
- **Where the slowness lives**: not in "re-reading a file" (that's IO, microseconds, negligible). The model must **re-process every context token each turn** (technically: prefill — all input tokens are computed before generating a reply). resume_json is a big blob of tokens; every turn's prefill recomputes it → slow, expensive, and it dilutes the attention weight of the system instructions (the PHASE 2 "going off the rails" is collateral).
- **Why session state is the cure**: once resume_json moves from "conversation history" to state, it **never enters context, is never sent to the model, never participates in prefill**. When a tool needs it, **code** fetches it from state (microseconds, no model involved). Smaller context → faster, cheaper, no drift, undiluted instructions. Four birds, one stone.

### Concept: there are two Geminis in this system (two layers) — who actually makes the suggestions
- **The main agent (orchestrator, agent.py)**: decides "which tool to call", talks to the user. **Cannot see** resume_json (it's in state).
- **The ephemeral Gemini inside each tool (e.g. inside analyze_jd_match)**: actually analyzes the resume and makes suggestions. **Can see** resume_json — because the tool code deliberately `json.dumps(resume_json)` into its prompt; used once and discarded.
- Conclusion: improvement suggestions are **not made by the main agent** — they come from the ephemeral Gemini inside analyze_jd_match that was fed resume_json + an internal prompt. → This explains all the earlier weirdness (summary suggestions, the C-project self-rationalization): fix the **tool's internal prompt**, not the orchestrator instruction.
- It also explains how the model "can still use" the resume in state: **the model doesn't understand it at all** — **code** does (Jinja2 rendering, rule checks, stuffing it into internal prompts). LLM does decisions (orchestration); code does deterministic work — the core division of labor in agent design.
- Read vs write-back: read-only tools (analyze/check/render/save) just take it and use it; only mutating operations (rewrite) need "fetch → modify → write back to state" — that's phase 2.

## 2026-06-04 — session state phase 2: rewrite write-back + deterministic schema-drift backstop

### Phase 2: rewrite_bullet writes back to state
- **Problem**: in phase 1, rewrite was still a pure function (bullet in, bullet out); the improved sentences never entered the resume in state → render/save kept showing old bullets. "Edited into the void."
- **Change (option A, direct write-back)**:
  - Signature: `original_bullet` → `bullet_id` + `tool_context`. The original sentence is **read from state by id** — the model no longer passes bullet text, closing another drift path (same thinking as phase 1).
  - New `_find_bullet()` scans bullets in both `experience[]` and `projects[]`.
  - The generate–validate loop is untouched; after editing, `bullet["content"]=new_bullet` then **reassign `state["resume_json"]=resume_json`** to force ADK to record the delta (mutating a nested dict alone may not persist; same single-write-point pattern as parse).
  - Backstops: no resume / unknown id → graceful error; already passing → attempts=0, no write; all 3 retries fail → write back the best attempt + warning (wording improvements aren't lost).
- **Why A over B (staging + explicit apply)**:
  - Performance/drift are **not** the deciding factors — both keep the resume away from the model. State IO is code-level microseconds, no prefill; write-back is **append-only history, never rewriting**: the full resume enters context exactly once (in parse's return) and is prompt-cache-friendly afterwards. The real old snowball was "the model repeatedly emitting the resume as function-call arguments" — write-back eliminates exactly that.
  - With performance tied, the only remaining question is the product one: "do we need a preview gate before persisting?" A welds "generate" and "persist" into one atomic call → no true preview (the user sees it after the change). B's staging gives preview but adds an apply tool + pending state. Conclusion: **design cost and UX are both about equal → chose A; the gate is the PHASE 3 checkbox moment**.
- **Verification** (asyncio simulating adk web + fake tool_context + real Gemini): _find_bullet across both sections / miss; no resume → error; bad id → error; already-passing → attempts=0, no write; weak bullet genuinely rewritten and written back to state. All green.

### Schema drift: upgraded from "probabilistic suppression" to "code guarantee"
- **Re-verified, conclusion first**: there is currently **no structural drift**. The parse prompt is hardened (explicitly bans `title`/`institution`/`item`/`text`), the template reads only canonical keys (`.content`/`.school`/`.name`/`.tech_stack`), rewrite reads `content` — all three contracts agree. The old DEVLOG todo predates the prompt hardening; superseded.
- **Residual risk**: parse output keys are still generated probabilistically by Gemini. Write-back makes rewrite **hard-depend on `content`** (drift to `text` → original reads empty → rewriting an empty string) — now the only place drift can really bite.
- **Fix**: after parse's `json.loads`, run `_normalize_schema()` — section-targeted renames (`institution→school`, `title→name`, `text/item/description→content`), **no blind recursion** to avoid clobbering same-named keys in skills/links. Even if it drifts, it lands back on canonical. Deterministic.
- Verified: drifted inputs all normalized; canonical inputs untouched; empty/missing sections don't crash. ✅

## 2026-06-07 (afternoon) — frontend PDF upload hang + parse_resume's pdf branch is dead code / a trap

### Symptom
After uploading a PDF in the Streamlit frontend (frontend/app.py), "Parsing resume…" spins forever.

### Diagnosis (isolation experiments were the key)
- Sent a **short text** message through the same Runner → returned 1 event normally in 7.6s. → **Rules out** a Streamlit↔ADK asyncio deadlock and rules out a broken frontend/backend connection.
- While hung: process at **0% CPU, zero model requests, no traceback** = waiting for the model to emit something enormous, not computing.
- **Root cause**: the frontend stuffed the whole PDF as base64 into one text message (app.py:118-121); to call `parse_resume(raw_text=<entire base64>)` the orchestrator must "recite" tens of thousands of base64 tokens into a function-call argument → extremely slow / over the output limit / MALFORMED. **The old "big data passing through the LLM" disease** (resume_json was cured with session state, but the PDF entrance still rammed straight through the model).

### Corollary: parse_resume's `source_format=='pdf'` branch
- In the whole repo, **only frontend app.py:121 uses `source_format='pdf'`**; nowhere else; in adk web the LLM defaults to text.
- I.e. **the only path that ever triggers this branch is the hanging one above.**
- The docstring even says "pass base64 for pdf" (parse_resume.py:45-47) → **actively luring the LLM into accepting base64** — a trap, not just dead code.

### Decision (direction A, **not yet implemented**, tomorrow)
- Frontend extracts text locally with pdfplumber and sends only text (the verified text path).
- Cut `parse_resume` down to a text-only tool: **delete the pdf branch + remove base64 from the docstring**, eliminating the "base64 through the LLM" trap.
- No other backend tools touched.

### Process lesson (note to self)
- When proposing the "frontend extracts text" fix, I should have **stated its downstream consequences in the same breath**: the pdf branch becomes dead code + the docstring would keep steering the LLM back into the pit. Supplying that only when the user pressed = failing to tell the full consequence of a change up front. Next time a proposal = also state what it kills / orphans / affects.

### Lessons

### Pitfall: missing dates slip through the whole chain ("the agent never asks for project dates and passes anyway")
- Symptom: a project has no dates, the agent never asks, and both rewrite and check_formatting wave it through.
- Triple root cause: ① rewrite only governs bullet prose — dates are not its job (innocent); ② check_formatting's date rule was `if date_value and …` — **format is only checked when a date exists; missing dates are silently skipped**, and warnings never flip `passed`; ③ the PHASE 2 clarification prompts all steer toward quantification — nobody owns "fill in missing basic fields".

### Pitfall: the GitHub link is "stored correctly but invisible"
- Symptom: user gives their GitHub URL, agent says it's added, the resume shows just the word "GitHub".
- Truth: the agent stored it right all along (the full URL is in the href) — **the template hardcoded the anchor text to "GitHub"/"LinkedIn"**. In print / ATS contexts the link effectively doesn't exist.
- Fix: the template now displays the real URL (stripping `https://`, `www.`, trailing slash); href unchanged, still clickable.
- Lesson: when a bug report says "the agent failed", first separate **data-layer not stored vs presentation-layer not shown** — this was the latter; prompt and tools were innocent.

### load_resume_version write-back to state ("saved but can't take it out" fixed)
- The user's pointed question: if I can only view and compare but never pull a version back out to edit/download, what's the database for? — exactly the gap recorded earlier (docstring had been made honest; behavior unfixed).
- Fix (**dual mode**): called as a tool by the agent (ADK injects ctx) → resume_json is written back to state as the working draft, response carries only metadata + `restored: True` (**the resume itself stays out of the response** — never dump the whole thing into context, the usual snowball rule); called from code (compare_versions passes no ctx) → behavior unchanged, returns the data, **state untouched** (comparing old versions never clobbers the draft).
- Safeguards: docstring + instruction state plainly "this REPLACES the current draft — confirm with the user if there are unsaved changes".
- Verified against real Atlas, 4/4 (tool mode writes state + slim response / code mode returns data / compare regression / bad id graceful).
- The database loop is now closed: save → list → **reload and keep working** → compare → download. Versioning is finally a real feature.

### Feature: a ⬇ icon next to each version in Version history for direct PDF download
- Zero LLM in the whole path: frontend calls load directly (code mode, draft untouched) → feeds a throwaway ctx to render_template for the PDF → bytes cached in session_state → the button becomes a real download in place.
- The two-step flow (click once to render, once to save) is a hard Streamlit constraint — `download_button` needs the bytes up front; render-on-demand + cache beats pre-rendering everything on refresh (20 versions × 2s = 40s, unacceptable).

### Deployment checklist (after the demo / before deploying)
- **Multi-user isolation**: the schema has had a `user_session` field since Task 2, but the whole chain hardcodes `"default"`; the frontend already has a per-session uuid — only the wiring is missing. Proper fix: user_session goes through session state (frontend writes it at session creation; tools read it from ctx.state; Gemini never fills this parameter). ### Pitfall: frequent "(no text response)" — old aiohttp blowing up genai's retry logic
- Symptom: right before the demo, the agent frequently returned nothing for a whole turn; the frontend fallback showed "(no text response)". 6 occurrences in one session log.
- Root cause: `google-genai`'s network retry logic references `aiohttp.ClientConnectorDNSError` (**added in aiohttp 3.11**), but the environment had 3.10.5 → every ordinary network blip (normally auto-retried) made the retry handler itself crash with `AttributeError` → the model call failed hard, zero reply.
- Fix: `python -m pip install --upgrade aiohttp` (3.10.5 → 3.14.1).
- Lesson: **an exact replay of this log's very first entry (service_identity)** — a traceback inside Google's library ≠ Google's fault; check dependency versions first. This bug lies dormant and only fires on network jitter, so "it worked fine before" does not rule out a dependency problem.
